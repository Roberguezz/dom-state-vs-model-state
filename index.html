Un laboratorio visual diseñado para entender de forma empírica los riesgos de almacenar la lógica en el árbol de nodos HTML (DOM).

01. Escala del Componente de Interfaz (HUD)

Simula una interfaz de juego o cuadro de mandos responsivo. Modifica el tamaño del HUD con el deslizador (limitado a un máximo de 300px). El sistema acoplado calcula el progreso basándose de forma absoluta en píxeles de pantalla, por lo que al estirar la interfaz, la barra nunca logra llenarse del todo. El sistema MVC se adapta perfectamente usando porcentajes nativos independientes de la escala física.

100px
Código Acoplado (DOM)
Nivel HUD: 1
Código Desacoplado (MVC)
Nivel HUD: 1
🔍 Explicación técnica y Código Fuente
no-mvc.js
// Lee los píxeles directamente de la pantalla
let px = parseInt(bar.style.width) || 0;
let nuevo = px + 20;

// ERROR: Si el HUD se escala a 300px, la barra crecerá sumando píxeles pero la condición estricta (>=100px) se romperá o reseteará antes de tiempo.
if (nuevo >= 100) {
    nivel.innerText = parseInt(nivel.innerText) + 1;
    nuevo = 0;
}
bar.style.width = nuevo + "px";
mvc.js
// Opera siempre sobre datos puros en memoria RAM
modelo.progreso += 20;

if (modelo.progreso >= 100) {
    modelo.nivel += 1;
    modelo.progreso = 0;
}

// La vista se limita a renderizar el porcentaje. No importa el tamaño del contenedor, el cálculo es exacto.
bar.style.width = modelo.progreso + "%";

¿Qué falla en el código acoplado? Al cambiar el tamaño del HUD a 300px, el código sigue asumiendo de manera fija que el límite es 100px de ancho. La lógica matemática quedó ligada a las propiedades de renderizado del navegador, alterando las condiciones del negocio al cambiar la resolución.

02. Conversión Dinámica de Divisas

Al cambiar de moneda, el checkout acoplado raspa el texto de la pantalla, pero al encontrar nuevos símbolos como el signo de dólar ($) o yenes (¥), el limpiador de texto se rompe dejando el total en un error irreversible (NaN) o congelando el precio. El sistema MVC procesa los datos numéricos de forma limpia en memoria.

Código Acoplado (DOM)
Total: 0.00€
Código Desacoplado (MVC)
Total: 0.00€
🔍 Explicación técnica y Código Fuente
no-mvc.js
// Intenta leer el texto directo del HTML
let texto = totalHtml.innerText; 

// BUG: Al cambiar a USD el texto pasa a "$11.00". El replace("€") no quita el "$", provocando que parseFloat devuelva un error NaN destructivo.
let actual = parseFloat(texto.replace("€", ""));
let nuevo = actual + 10.00;

totalHtml.innerText = nuevo + "€";
mvc.js
// Modifica el flotante puro en memoria JS
modelo.eurosBase += 10.00;

// La vista se limita a multiplicar y formatear
let tasa = listaTasas[divisaActiva];
let totalReal = modelo.eurosBase * tasa;

totalHtml.innerText = formatearTexto(totalReal, divisaActiva);

¿Qué falló en el código acoplado? Se usó la interfaz de usuario como si fuera una base de datos de texto (técnica conocida como Data Scraping). En cuanto la interfaz cambió su formato internacional para mostrar el símbolo en otra posición, las funciones de limpieza de texto fallaron e inyectaron un valor corrupto NaN en las cuentas de cobro.

03. Ciclo de Vida y Persistencia de Formulario

Al avanzar en un formulario por pasos, las aplicaciones modernas eliminan del DOM los campos ocultos para ahorrar memoria. Escribe tu nombre en el Paso 1, avanza al Paso 2 y luego regresa. En el panel acoplado notarás que tu nombre se ha borrado para siempre al desaparecer su etiqueta HTML. En el panel MVC, el dato permanece a salvo en el modelo.

Código Acoplado (DOM)
Payload capturado: null
Código Desacoplado (MVC)
Payload capturado: null
🔍 Explicación técnica y Código Fuente
no-mvc.js
// Reconstruye el HTML destruyendo el anterior
function cambiarAPaso2() {
    // Al sobreescribir el contenedor con innerHTML, el valor que escribió el usuario en el Paso 1 se elimina de la existencia por completo.
    contenedor.innerHTML = HTML_PASO_2;
}
mvc.js
// Captura los datos en tiempo real al escribir
contenedor.addEventListener("input", (e) => {
    modelo.formulario[e.target.name] = e.target.value;
});

// Al regresar al Paso 1, la vista lee la memoria
function renderPaso1() {
    contenedor.innerHTML = HTML_PASO_1;
    document.getElementById("inputNombre").value = modelo.formulario.nombre;
}

¿Por qué se borran los datos en el caso acoplado? Al cambiar de paso, el script utiliza innerHTML para limpiar la vista y dibujar los nuevos campos. Como el valor del campo de texto no se guardó en ninguna variable de JavaScript, la información desaparece al destruir el elemento HTML de la pantalla.
