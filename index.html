LABORATORIO ARQUITECTÓNICO

Separación de Estado

Un laboratorio visual diseñado para entender de forma empírica la diferencia de almacenar la lógica en el árbol de nodos HTML (DOM) frente a un flujo unificado y centralizado en memoria (MVC).

01 // ESCALABILIDAD RESPONSIVA

Escala del Componente de Interfaz (HUD)

Un consejo de colega: Imagina que estás maquetando un componente responsivo o un HUD de juego que escala según la pantalla. Si tu lógica calcula el progreso leyendo directamente los píxeles físicos del elemento en el DOM, tu aplicación se romperá al cambiar el tamaño de la interfaz. Al acoplar el cálculo matemático a los píxeles de pantalla, dependes del renderizado físico del navegador. Con MVC, mantenemos el estado puro en memoria con porcentajes y dejamos que la vista se adapte libremente.

100px
Código Acoplado (DOM)
Nivel HUD: 1
Código Desacoplado (MVC)
Nivel HUD: 1
Análisis Técnico y Código de Muestra
no-mvc.js
// Lee los píxeles directamente del estilo de la pantalla
let px = parseInt(barCoupled.style.width) || 0;
let nuevo = px + 20;

// Si el HUD cambia a 300px, el límite estricto de 100px rompe el flujo
if (nuevo >= 100) {
    lvlCoupled.innerText = parseInt(lvlCoupled.innerText) + 1;
    nuevo = 0;
}
barCoupled.style.width = nuevo + "px";
mvc.js
// Modifica el dato puro en memoria (en porcentaje)
appState.s1.progreso += 20;

if (appState.s1.progreso >= 100) {
    appState.s1.nivel += 1;
    appState.s1.progreso = 0;
}

// La vista renderiza el porcentaje adaptativo de forma reactiva
barMvc.style.width = appState.s1.progreso + "%";

¿Por qué falla el acoplamiento? Al cambiar el tamaño del HUD a 300px, el código acoplado sigue limitando de forma rígida la barra a 100px, debido a que su lógica matemática quedó ligada a las dimensiones físicas de la pantalla. Separando el estado (MVC), el cálculo es inmune a las dimensiones visuales del contenedor.

02 // INTEGRIDAD DE DATOS

Conversión Dinámica de Divisas

Un consejo de colega: Es muy tentador hacer "data scraping" leyendo el texto de las etiquetas HTML para ahorrarte una variable en JS. Pero mira lo que ocurre al internacionalizar tu app: en cuanto aparece un símbolo de otra divisa como el signo de dólar ($) o yenes (¥), tus funciones de limpieza de texto fallan estrepitosamente, inyectando un valor corrupto (NaN) de forma irreversible en tus cuentas. Si mantienes el valor numérico limpio en el modelo, la vista solo se limita a presentarlo formateado. ¡Tus cuentas saldrán perfectas!

Código Acoplado (DOM)
0.00 €
Código Desacoplado (MVC)
0.00 €
Análisis Técnico y Código de Muestra
no-mvc.js
// Lee el texto visible directo de la pantalla
let texto = totalCoupled.innerText;

// BUG: Al cambiar a USD el texto pasa a "$ 11.00".
// El replace(" €") no quita el "$" y el parser inyecta un NaN destructivo
let limpio = parseFloat(texto.replace(" €", ""));
let total = limpio + 10.00;

totalCoupled.innerText = total + " €";
mvc.js
// El incremento financiero ocurre sobre memoria RAM pura
appState.s2.saldoEuros += 10.00;

// La vista se limita a multiplicar por la tasa y dar formato
let tasa = appState.tasas[divisaActiva];
let totalCalculado = appState.s2.saldoEuros * tasa;

totalMvc.innerText = formatear(totalCalculado, divisaActiva);

¿Por qué ocurre este error? Se ha usado la interfaz como si fuese una base de datos (técnica frágil llamada Data Scraping). Al alterar el orden o los símbolos de los caracteres monetarios internacionales, las expresiones regulares básicas fallan y devuelven un estado corrupto NaN.

03 // CICLO DE VIDA DEL DOM

Persistencia de Formulario y Ciclo de Vida

Un consejo de colega: En aplicaciones interactivas modernas por pasos, solemos destruir y renderizar partes del DOM para mantener la interfaz ligera y optimizada. Si confías en que los inputs del HTML almacenen la información del usuario, al eliminar la sección para avanzar de pantalla, esos datos se evaporarán. Centralizar los inputs en un modelo en memoria garantiza que, aunque destruyas y recrees la interfaz mil veces, la información de tu usuario estará sana y salva. Escribe tu nombre en el paso 1, avanza al paso 2 y luego regresa: verás que la versión acoplada te ha borrado el nombre para siempre.

Payload capturado: null
Código Desacoplado (MVC)

Análisis Técnico y Código de Muestra
no-mvc.js
// Reemplaza físicamente la vista con innerHTML
function irAPaso2() {
    // Al sobrescribir el nodo padre, el valor del input del Paso 1
    // que el usuario escribió se destruye por completo del DOM
    formContainerCoupled.innerHTML = HTML_STEP_2;
}
mvc.js
// Escucha los cambios del input en tiempo real sobre el Modelo
formContainerMvc.addEventListener("input", (e) => {
    appState.s3.formulario[e.target.name] = e.target.value;
});

// Al reconstruir los campos, se restauran desde memoria
function renderPaso1() {
    formContainerMvc.innerHTML = HTML_STEP_1;
    document.getElementById("inputNombreMvc").value = appState.s3.formulario.nombre;
}

¿Por qué se pierde la información? Al usar innerHTML para cambiar de pantalla, el navegador destruye los nodos antiguos del DOM. Puesto que el valor escrito no existía en variables de JavaScript, el estado de los datos del usuario es destruido permanentemente al desaparecer su representación visual.

04 // SINCRONIZACIÓN MULTIVISTA

Sincronización Multivista en Tiempo Real

Un consejo de colega: Imagina que tienes un carrito de compras y necesitas actualizar tres vistas independientes al mismo tiempo: un contador del carrito en la cabecera, la lista detallada de servicios en el centro y el total acumulado en el pie de página. Con el código acoplado al DOM, cada click te obliga a buscar y mutar a mano cada uno de los elementos visuales por separado. Si mañana añades una cuarta vista, tu código se volverá increíblemente frágil. Con MVC, modifiques lo que modifiques, solo actualizas una colección en memoria y la función de renderizado se encarga de que todas las vistas permanezcan unificadas y sin fallas.

Análisis Técnico y Código de Muestra
no-mvc.js
// Inserción manual a múltiples elementos del DOM
cartBadgeCoupled.innerText = coupledCartItems.length;

const li = document.createElement("li");
li.innerHTML = `${newItem.name}$${newItem.price}`;
cartListCoupled.appendChild(li);

let total = 0;
coupledCartItems.forEach(item => total += item.price);
cartTotalCoupled.innerText = `$ ${total.toFixed(2)}`;
mvc.js
// 1. Mutación exclusiva de la estructura en memoria
appState.s4.items.push(nuevoItem);

// 2. Renderizado unificado y consistente
function renderizar() {
    cartBadgeMvc.innerText = appState.s4.items.length;
    cartListMvc.innerHTML = appState.s4.items.map(renderItemHtml).join("");
    cartTotalMvc.innerText = `$ ${calcularTotal().toFixed(2)}`;
}

¿Qué ventaja aporta el desacoplamiento? Con el código acoplado, actualizar 3 vistas independientes requiere escribir selectores imperativos redundantes para cada elemento en el DOM. En cambio, con MVC la interfaz se reduce a ser un reflejo directo del Modelo. Si agregamos una cuarta o quinta vista en el futuro, no es necesario tocar los controladores existentes, garantizando consistencia absoluta y escalabilidad.
