Patrones de Arquitectura

Separación de Vista y Modelo

Ejemplos prácticos e interactivos que demuestran por qué mantener el estado de la aplicación en el DOM dificulta el mantenimiento, y cómo delegar la lógica de negocio a un Modelo en memoria soluciona este problema de diseño.

01 // DIMENSIONAMIENTO DEL CONTENEDOR

Escala Física del Contenedor

Cuando se cambia el tamaño de un componente de interfaz en el DOM, es un error de diseño calcular su progreso leyendo el ancho físico del contenedor en píxeles. Si el contenedor se redimensiona, la lógica matemática ligada a las dimensiones físicas de pantalla se rompe. Con una vista desacoplada, mantenemos el progreso en un porcentaje matemático puro en memoria (Modelo) y dejamos que el DOM se limite a pintarlo de forma responsiva.

100px
Código Acoplado (Lectura del DOM)
Nivel actual: 1
Código Desacoplado (Modelo en memoria)
Nivel actual: 1
Código fuente de comparación
no-mvc.js
// Lee los píxeles directamente del estilo de la pantalla
let px = parseInt(barCoupled.style.width) || 0;
let nuevo = px + 20;

// Si el contenedor se redimensiona a 300px, el límite estricto de 100px falla
if (nuevo >= 100) {
    lvlCoupled.innerText = parseInt(lvlCoupled.innerText) + 1;
    nuevo = 0;
}
barCoupled.style.width = nuevo + "px";
mvc.js
// Modifica el dato puro en memoria en porcentaje (Modelo)
appState.s1.progreso += 20;

if (appState.s1.progreso >= 100) {
    appState.s1.nivel += 1;
    appState.s1.progreso = 0;
}

// La vista se limita a renderizar el porcentaje de forma reactiva
barMvc.style.width = appState.s1.progreso + "%";

Explicación: Al redimensionar el contenedor a 300px, el código acoplado mantiene el límite rígido a 100px debido a que su lógica matemática depende de los píxeles físicos de renderizado de la pantalla. Al separar el estado en un Modelo en memoria, el cálculo matemático se independiza de las dimensiones visuales del elemento.

02 // INTEGRIDAD DE DATOS

Conversión Dinámica de Divisas

Intentar recuperar valores leyendo el texto visible en las etiquetas del DOM es una práctica frágil. Al dar formato a las monedas o cambiar a divisas como dólares ($) o yenes (¥), las funciones de parseo y limpieza de texto fallan, inyectando un valor de número inválido (NaN). Mantener el estado financiero puro en el Modelo garantiza que las sumas y las conversiones siempre operen sobre números correctos.

Código Acoplado (Lectura del DOM)
0.00 €
Código Desacoplado (Modelo en memoria)
0.00 €
Código fuente de comparación
no-mvc.js
// Lee el texto visible directamente de la pantalla
let texto = totalCoupled.innerText;

// BUG: Al cambiar a USD ("$ 11.00"), la limpieza de "€" no surte efecto.
// parseFloat falla en su análisis devolviendo NaN, corrompiendo la suma.
let limpio = parseFloat(texto.replace(" €", ""));
let total = limpio + 10.00;

totalCoupled.innerText = total + " €";
mvc.js
// El incremento se realiza sobre un número puro en memoria
appState.s2.saldoEuros += 10.00;

// La vista solo se encarga de formatear el valor a partir de los datos
let tasa = appState.tasas[divisaActiva];
let totalCalculado = appState.s2.saldoEuros * tasa;

totalMvc.innerText = formatear(totalCalculado, divisaActiva);

Explicación: Al extraer el estado directamente de la interfaz, cualquier variación en el orden o en el tipo de caracteres de los símbolos internacionales provoca fallos en las expresiones regulares básicas, resultando en estados corruptos persistentes.

03 // CICLO DE VIDA DE NODOS

Persistencia e inputs Dinámicos

Al alternar o recrear secciones HTML dinámicamente en aplicaciones frontend, confiar el estado a los inputs del DOM provoca la pérdida irreversible de los datos en cuanto el nodo se desmonta o destruye del árbol del navegador. Centralizar las entradas en un Modelo en memoria asegura que la información permanezca persistente e independiente del ciclo de vida o renderizado de la interfaz. Escribe un nombre en el paso 1, avanza al paso 2 y regresa: el input acoplado se habrá borrado.

Código Acoplado (Lectura del DOM)
Datos capturados: null
Código Desacoplado (Modelo en memoria)
Datos capturados: null
Código fuente de comparación
no-mvc.js
// Cambia la sección reescribiendo el contenido del contenedor
function irAPaso2() {
    // Al sobrescribir el contenedor, los elementos HTML antiguos
    // y los datos escritos en ellos se eliminan de la memoria.
    formContainerCoupled.innerHTML = HTML_STEP_2;
}
mvc.js
// Sincroniza la entrada del input dinámicamente con el Modelo
formContainerMvc.addEventListener("input", (e) => {
    appState.s3.formulario[e.target.name] = e.target.value;
});

// Al reconstruir los inputs, los valores se inyectan desde el Modelo
function renderPaso1() {
    formContainerMvc.innerHTML = HTML_STEP_1;
    document.getElementById("inputNombreMvc").value = appState.s3.formulario.nombre;
}

Explicación: Modificar contenedores mediante innerHTML destruye los objetos de nodo anteriores. Si los datos ingresados no se guardan en variables fuera del DOM, la información del usuario se pierde definitivamente al desmontar la interfaz visual.

04 // COMBATE RPG Y MULTIPLICADOR DE COMBO

Combate RPG y Multiplicador de Combo

Al encadenar golpes rápidos, el sistema activa un multiplicador de combo que escribe en el marcador el texto "¡COMBO! 120". En la versión acoplada al DOM, el código recupera el total acumulado leyendo ese texto de la pantalla e intenta parsearlo numéricamente. Al procesar una cadena con letras, parseFloat devuelve NaN, corrompiendo el marcador de forma irrecuperable. Con el Modelo en memoria, el acumulado numérico permanece intacto independientemente de cómo lo muestre la vista.

Código Acoplado (Lectura del DOM)
(*'-'*)
0
Código Desacoplado (Modelo en memoria)
(*'-'*)
0
Código fuente de comparación
no-mvc.js
// Lee el marcador de daño directo de la pantalla
let textVal = totalDmgCoupled.innerText;
let currentDmg = parseFloat(textVal); // ¡COMBO! 120 -> NaN

let baseDamage = 20;
let addedDamage = comboActive ? (baseDamage * 2) : baseDamage;

// Escribe el string formateado directamente en la pantalla
totalDmgCoupled.innerText = `¡COMBO! ${currentDmg + addedDamage}`;
mvc.js
// Mutamos la variable numérica en memoria de forma segura
appState.s4.damage += hitDamage;
appState.s4.isCombo = comboActive;

// La vista se limita a renderizar el estado del Modelo
if (appState.s4.isCombo) {
    totalDmgMvc.innerText = `¡COMBO! ${appState.s4.damage}`;
} else {
    totalDmgMvc.innerText = appState.s4.damage;
}

Explicación: En la versión acoplada, al encadenar golpes rápidos, el DOM recibe texto decorativo ("¡COMBO! 80"). En el siguiente ataque, el código intenta recuperar el acumulado leyendo el elemento del DOM y parseándolo, provocando un error matemático irreversible NaN. Al separar el estado en un Modelo numérico puro, la lógica de cálculo permanece a salvo del renderizado y el flujo nunca se interrumpe.

05 // ACUMULACIÓN DE TEMPORIZADORES ASÍNCRONOS

Timer Stacking: Temporizadores Apilados

Al iniciar bucles asíncronos con setInterval, es imprescindible guardar la referencia del identificador de temporizador para poder cancelarlo antes de crear uno nuevo. En la versión acoplada al DOM no existe un Modelo que almacene dicha referencia, por lo que cada pulsación del botón o cada cambio en el deslizador de velocidad apila un nuevo bucle sobre los anteriores. El resultado es inmediatamente visible: el contador se dispara de forma exponencial. En la versión MVC, el identificador se guarda en el Modelo, lo que permite limpiar el bucle previo antes de arrancar uno nuevo, manteniendo siempre exactamente un temporizador activo.

Código Acoplado (Lectura del DOM)
800ms
Ticks: 0
Código Desacoplado (Modelo en memoria)
800ms
Ticks: 0
Código fuente de comparación
no-mvc.js
// No existe referencia al timer anterior; cada llamada apila uno nuevo
btnS5StartCoupled.addEventListener("click", () => {
    const ms = parseInt(sliderS5Coupled.value);

    // Sin clearInterval previo: los timers se acumulan indefinidamente
    setInterval(() => {
        let n = parseInt(tickCountCoupled.innerText) || 0;
        tickCountCoupled.innerText = n + 1;
        tickIndicatorCoupled.classList.toggle("tick-active");
    }, ms);
});
mvc.js
btnS5StartMvc.addEventListener("click", () => {
    // Limpia el timer anterior antes de crear uno nuevo
    clearInterval(appState.s5.timerId);

    appState.s5.interval = parseInt(sliderS5Mvc.value);
    appState.s5.timerId = setInterval(() => {
        appState.s5.ticks++;
        renderizar(); // La vista refleja el Modelo
    }, appState.s5.interval);
});

Explicación: En la versión acoplada, la referencia al identificador de temporizador no se almacena en ningún lugar externo al DOM, por lo que es imposible cancelar el bucle previo antes de crear uno nuevo. Cada pulsación de "Iniciar" añade un bucle paralelo. En la versión MVC, el identificador se guarda en appState.s5.timerId y se cancela explícitamente antes de arrancar el nuevo ciclo, garantizando que solo exista un temporizador activo en cualquier momento.

06 // GRADIENTE DE COLOR DINÁMICO

Selector de Colores y Gradientes en Vivo

Cambiar colores en la interfaz se ve simple, pero si el estado se almacena solo en el DOM (por ejemplo, en un atributo style o en valores de input sin sincronización), modificar el contenedor puede perder la información. En la versión acoplada, al resetear la página o cambiar de sección, el color seleccionado se pierde. Con un Modelo centralizado, cualquier cambio de color se mantiene persistente incluso después de recargas del DOM.

Código Acoplado (Lectura del DOM)
Color seleccionado: #ff6b6b
Código Desacoplado (Modelo en memoria)
Color seleccionado: #6bcf7f
Código fuente de comparación
no-mvc.js
colorPickerCoupled.addEventListener("change", (e) => {
    const color = e.target.value;
    // El gradiente se aplica directamente al DOM sin sincronización
    const gradient = `linear-gradient(135deg, ${color}, #ffa500)`;
    gradientBoxCoupled.style.background = gradient;
    colorDisplayCoupled.innerText = color;
    // Si el contenedor se reconstruye, el color se pierde
});
mvc.js
colorPickerMvc.addEventListener("change", (e) => {
    // El color se sincroniza primero en el Modelo
    appState.s6.color = e.target.value;
    renderizarS6();
});

function renderizarS6() {
    const gradient = `linear-gradient(135deg, ${appState.s6.color}, #4ec9b0)`;
    gradientBoxMvc.style.background = gradient;
    colorDisplayMvc.innerText = appState.s6.color;
}

Explicación: Sin un Modelo centralizado, cambios visuales como gradientes y colores quedan atrapados en el estilo inline del DOM. Si la interfaz se reconstruye o se resetea, esa información desaparece. Centralizar el estado en memoria garantiza que cualquier mutación visual sea reversible y recuperable.

07 // INTERFAZ VISUAL COMPLEJA: LISTA EDITABLE

Gestión de Lista Editable con Sincronización

Construir interfaces dinámicas con listas que se pueden añadir, eliminar y editar es complicado sin un Modelo. En la versión acoplada, si intentas recuperar los elementos de la lista leyendo el DOM, cambios en el HTML pueden perder datos. Con el Modelo en memoria, cada elemento de la lista está sincronizado con su representación visual, permitiendo operaciones seguras como edición inline, eliminación y reordenamiento sin corrupción de datos.

Código Acoplado (Lectura del DOM)
Elementos: 0
Código Desacoplado (Modelo en memoria)
Elementos: 0
Código fuente de comparación
no-mvc.js
btnAddListCoupled.addEventListener("click", () => {
    const texto = inputListCoupled.value;
    if (!texto) return;
    
    // Añade directamente al DOM sin sincronización
    const li = document.createElement("li");
    li.innerHTML = `${texto} 
        <button onclick="this.parentNode.remove()">×</button>`;
    listContainerCoupled.appendChild(li);
    
    // El contador se calcula leyendo el DOM cada vez
    countCoupled.innerText = listContainerCoupled.children.length;
    inputListCoupled.value = "";
});
mvc.js
btnAddListMvc.addEventListener("click", () => {
    const texto = inputListMvc.value;
    if (!texto) return;
    
    // Añade primero al Modelo
    appState.s7.items.push({ id: Date.now(), text: texto });
    renderizarListaMvc();
    inputListMvc.value = "";
});

function renderizarListaMvc() {
    listContainerMvc.innerHTML = appState.s7.items
        .map(item => `<li>${item.text} 
            <button data-id="${item.id}">×</button></li>`)
        .join("");
    countMvc.innerText = appState.s7.items.length;
}

Explicación: Manipular directamente el DOM sin un Modelo centralizado causa problemas cuando necesitas realizar búsquedas, filtrados o ediciones. Cada operación requiere leer del DOM, lo que es frágil y propenso a errores. Con un Modelo, todas las operaciones son seguras y predecibles.
